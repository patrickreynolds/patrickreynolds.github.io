Enumerables and why they are so C@@L
May 17, 2014 - 2:02pm
Ruby Docs: The Enumerable mixin provides collection classes with several traversal and searching methods, and with the ability to sort. The class must provide a method each, which yields successive members of the collection. If Enumerable#max, #min, or #sort is used, the objects in the collection must also implement a meaningful <=> operator, as these methods rely on an ordering between members of the collection.

For this review, I have taken an in-depth look into the Enumerable#group_by function.This is a surprizingly strong function with quite a bit behind the hood and very little syntax for the user. The group_by function in short takes in an array of objects, and performs an operation on them, and then returns a hash of inputs and their results grouped by the result of the applied operation. That might be quite a bit to wrap your head around at first, but let me explain through example.

Take 1, 2, 3, 4, 5, 6 as inputs.

Let's say we want to use the group_by method to find all of the odd and even numbers in this given set. All we would have to do is run the following set:

(1..6).group_by{ | i | i % 2 }

The result of this would return a hash representing all results grouped by similar output as follows:

{1=>[1, 3, 5], 0=>[2, 4, 6]}

So, that being said, we can restate the line before with a bit more context. The group_by function in short takes in an array of objects, performs an operation on them (in this case modulous 2), and then returns a hash of inputs and their results grouped by the result of the applied operation.
